### Error Handling

Error handling is comprehensive and well-designed:

- **Signal interrupts**: `index.ts` correctly silently returns on `SignalInterruptError` / `PlanSignalInterruptError`.
- **Consolidation pipeline**: Each step is checkpointed via `consolidation-checkpoint.ts`; crashes resume from last completed step.
- **Rate limiting**: Global retry budget via `RateLimitRetryState` prevents runaway retries. Shared across all rounds and normal retries within an instance.
- **Windows filesystem**: EBUSY/EPERM/ENOTEMPTY retries in `cleanupTempDir()` with exponential backoff.
- **Claude failures**: Dedup throws on failure (critical path); hierarchy falls back to flat structure with debug logging (non-critical path); single-doc discovery falls back to raw content.
- **Instance failures**: Multi-level retry hierarchy: rate-limit retries (shared budget) -> normal retries (per-round) -> permanent failure marking.

   - These 3 branches would close the 5.72% gap entirely.

2. **Raise `instance-manager/rounds.ts` branch coverage above 95%.** Uncovered lines 176-182 are the `onFailure` callback and `RetryInfo` initialization when the initial spawn fails (before entering the retry loop). Lines 217 and 225 are within the retry loop (error accumulation and permanently-failed callback). A test that exercises the full "initial fail -> retry -> exhaust retries" path with all progress callbacks asserted would cover these.

### Nice to Have

3. **Consider a global test timeout.** Several integration tests run close to the default 5000ms timeout. A `testTimeout: 10000` in `vitest.config.ts` would prevent future timeout regressions as the module graph grows with more splits. The iteration 10 timeout regression was a warning that this could recur.

4. **Migrate `browser-open.ts` from `exec()` to `execFile()`.** This would eliminate the shell metacharacter risk entirely. The change is small (16-line file) and straightforward:
   ```typescript
   // Windows: execFile('cmd', ['/c', 'start', '""', filePath])
   // macOS:   execFile('open', [filePath])
   // Linux:   execFile('xdg-open', [filePath])
   ```
   This has been carried across multiple iterations as an accepted risk. Worth fixing since it's a 5-minute change.

5. **Consider a lightweight arg parsing library.** `cli.ts` at 413 lines is the second-largest top-level source file. The manual parsing works well, but `node:util parseArgs` (available since Node 18.3, no external dependency) could simplify it as new flags are added.

## Future Considerations

### Features and Improvements

- **Claude Agent SDK migration**: The most impactful architectural change available. Would replace subprocess spawning (`runClaude` in `claude-cli.ts`) with direct API calls, enabling shared context, token reuse, and finer lifecycle control. The clean `claude-cli.ts` interface (123 lines) makes this a well-bounded migration. The `ClaudeCliResult` type is used throughout and would need updating, but the clear separation of concerns makes this manageable.

- **Finding severity filtering (`--min-severity`)**: The data model already supports severity levels with a defined ranking in `consolidation/types.ts`. Implementation would add a filter step after consolidation. Straightforward.

- **Plan validation subcommand (`uxreview validate-plan`)**: The plan template format is well-defined (## headings, - bullets). A validation command could check format before expensive analysis runs.

- **Consolidation as a separate CLI subcommand**: Currently consolidation is tightly coupled to the orchestration flow. Extracting it as `uxreview consolidate` would allow re-running consolidation on existing instance outputs without re-running analysis.

- **AbortController for cancellation**: The current signal handler approach works but is bespoke. Using `AbortController` (standard Node.js API) would simplify cancellation propagation, especially if the Agent SDK migration happens.

### Architectural Decisions to Revisit

- **Two parallel orchestrator modules.** `orchestrator.ts` (415 lines) and `plan-orchestrator.ts` (269 lines) share structural patterns: workspace init, instance spawning, progress display, signal handling, cleanup. A base orchestrator class or composition pattern could reduce this duplication, though the current approach works well and both modules are reasonably sized after the instance-manager extraction.

- **File-based IPC.** The system uses file reads/writes for communication between the orchestrator and Claude instances (checkpoint.json, discovery.md, report.md). This works reliably and enables resumability, but limits observability and makes concurrent access harder to reason about. The Agent SDK's built-in communication would improve this.

- **Rate-limit retry budget scope.** The `RateLimitRetryState` is shared within a single instance's round execution. If multiple instances hit rate limits simultaneously, each has its own budget. A global rate-limit coordinator could be more efficient at backing off when the Claude API is under pressure.
